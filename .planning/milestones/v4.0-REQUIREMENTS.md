## Phase 17: 安全加固

### SEC-AUTH-01: 学生密码哈希化

- **描述：** 将 `students.password` 从明文存储迁移到 bcrypt 哈希，消除密码泄露风险。
- **用户故事：**
  - 作为一名系统管理员，我希望学生密码以不可逆哈希形式存储，即使数据库泄露也无法还原明文密码。
  - 作为一名教师，创建学生账号时系统应生成安全的随机临时密码，而非所有人默认 123456。
- **验收标准：**
  1. `students.password` 列存储 bcrypt 哈希值（cost factor ≥ 10）
  2. 移除 `'123456'` 默认密码硬编码
  3. 新建学生时生成 8 位随机临时密码，通过 API 响应返回给教师
  4. 学生登录时使用 `bcrypt.compare()` 验证
  5. 密码修改接口使用 bcrypt 哈希新密码
  6. 密码长度 ≥ 8 位，必须包含字母和数字
- **影响文件：**
  - `packages/core/db/index.ts` — students 表默认值
  - `server.ts` — 学生创建/更新/登录路由
  - `packages/plugins/management.ts` — 学生管理命令
- **风险：** 需安装 `bcrypt` 或 `@node-rs/bcrypt` 原生依赖

### SEC-AUTH-02: 教师/管理员密码升级

- **描述：** 将教师和管理员的 SHA-256 无盐哈希平滑迁移到 bcrypt。
- **用户故事：**
  - 作为一名管理员，我不希望我的密码因为使用弱哈希算法而能被彩虹表破解。
- **验收标准：**
  1. 新增 `users.password_hash` 列存储 bcrypt 哈希
  2. 保留旧 `users.password_hash` 作为过渡
  3. 登录时：优先用 bcrypt 验证；若为 SHA-256 哈希且验证通过，自动重新哈希为 bcrypt 并更新数据库
  4. 提供 CLI 脚本 `npm run migrate-passwords` 批量升级
  5. 种子用户（admin/admin、teacher/teacher）默认使用 bcrypt
- **影响文件：**
  - `packages/core/db/index.ts` — 种子用户创建
  - `server.ts` — 登录验证逻辑
- **依赖：** SEC-AUTH-01（共享 bcrypt 依赖）

### SEC-AUTH-03: Session 安全强化

- **描述：** 为 Session Token 添加过期机制、Secure 标志和用户关联。
- **用户故事：**
  - 作为一名安全审计员，我希望 session 有明确的过期时间，而不是永久有效。
  - 作为一名用户，我修改密码后所有已登录设备应被强制下线。
- **验收标准：**
  1. `Max-Age` 改为 24 小时（空闲超时），每次请求刷新
  2. 绝对过期 7 天（即使持续活跃），强制重新登录
  3. 生产环境设置 `Secure` cookie 标志
  4. `client_sessions` 表增加 `user_id` 和 `expires_at` 列
  5. 启动时清理过期 session（`DELETE FROM client_sessions WHERE expires_at < ?`）
  6. 密码修改后删除该用户所有 session
- **影响文件：**
  - `server.ts` — Set-Cookie、登录/登出路由
  - `packages/core/db/index.ts` — client_sessions 表 ALTER

### SEC-AUTH-04: 登录频率限制

- **描述：** 对登录端点实施频率限制，防止暴力破解。
- **验收标准：**
  1. `/api/auth/login` 限制为 5 次请求 / IP / 分钟
  2. 连续 3 次失败后，该 IP 需要等待 30 秒才能重试
  3. 失败计数器存储在内存（LRU cache），不依赖 Redis
  4. 超过阈值返回 HTTP 429 及 `Retry-After` 头
  5. 异常高频尝试记录 WARN 级别日志
- **影响文件：** `server.ts`（或新建 `server/middleware/rate-limit.ts`）
- **新依赖：** `express-rate-limit`

### SEC-AUTH-05: 学生自助密码修改

- **描述：** 为学生提供密码修改功能，无需教师介入。
- **验收标准：**
  1. `POST /api/auth/change-password` — 接受 `{ oldPassword, newPassword }`，验证旧密码后更新
  2. 前端学生面板增加"修改密码"入口
  3. 密码修改成功后，除当前 session 外所有该用户的 session 失效
- **影响文件：** `server.ts`、`src/App.tsx`

### SEC-DATA-01: AI Provider API Key 加密存储

- **描述：** 将 `ai_providers.api_key` 从明文改为 AES-256-GCM 加密存储。
- **验收标准：**
  1. `api_key` 列存储 Base64 编码的加密数据（格式：`{iv}:{authTag}:{ciphertext}`）
  2. 加密密钥通过环境变量 `ENCRYPTION_KEY` 注入（32 字节 hex）
  3. 若未配置 `ENCRYPTION_KEY`，服务启动失败并给出明确错误提示
  4. `GET /api/ai-providers` 返回的 api_key 被 mask：`sk-1a2b****c3d4`
  5. `PUT /api/ai-providers` 接收明文 key，加密后存储
  6. 解密仅在构造 AI API 请求的 `Authorization` 头时进行
- **影响文件：** `server.ts`（AI provider CRUD 路由）、`packages/core/di/ai-service.ts`

### SEC-DATA-02: 学生数据隐私合规

- **描述：** 实现学生数据导出和删除功能，满足 GDPR/CCPA 基本合规要求。
- **验收标准：**
  1. `GET /api/students/:id/export` — 返回 JSON，包含该学生所有个人数据（档案、提交、成绩、考勤、评语等）
  2. `DELETE /api/students/:id/gdpr-delete` — 完全删除学生所有数据（需管理员权限 + 二次确认）
  3. 删除操作写入审计日志
- **影响文件：** `server.ts`（新路由）

### SEC-DATA-03: 文件上传安全校验

- **描述：** 增加文件上传的 magic bytes 校验、大小限制和 zip bomb 防护。
- **验收标准：**
  1. 上传文件校验文件头魔数（前 4-8 字节）与声明类型匹配
  2. 拒绝可执行文件类型（.exe、.sh、.bat、.dll、.so 等）
  3. ZIP 上传计算解压后总大小，超过限制拒绝
  4. 大小限制：课件 50MB、插件 ZIP 10MB、学生作业 20MB
- **影响文件：** `server.ts` 上传路由、`server/utils/file-upload.ts`

### SEC-NET-01: CORS 白名单化

- **描述：** 将 CORS 从通配符改为白名单模式。
- **验收标准：**
  1. Socket.IO CORS origin 从 `'*'` 改为环境变量配置
  2. Express CORS 中间件同步配置
  3. 开发环境默认允许 `localhost:5173`、`localhost:9000`
  4. 生产环境仅允许 `ALLOWED_ORIGINS` 中列出的域名
- **影响文件：** `server.ts` Socket.IO 初始化

### SEC-NET-02: 安全 HTTP 头

- **描述：** 集成 helmet 添加标准安全响应头。
- **验收标准：**
  1. 使用 `helmet()` 默认配置
  2. 自定义 CSP：`script-src 'self' 'unsafe-inline'`（需适配 MFE 动态脚本）
  3. 添加 `Strict-Transport-Security: max-age=31536000`
  4. 添加 `X-Content-Type-Options: nosniff`
- **新依赖：** `helmet`

### SEC-NET-03: 输入验证与消毒

- **描述：** 统一使用 zod 进行请求体验证，AI 聊天消息进行 XSS 消毒。
- **验收标准：**
  1. 核心路由引入 zod schema 验证请求体
  2. AI 聊天消息中的 HTML 标签转义
  3. AI 输出中的 HTML 标签转义后再渲染到前端
  4. 验证失败返回 400 + 结构化错误信息
- **新依赖：** `xss`（或使用 DOMPurify 服务端版本）

### SEC-NET-04: AI Prompt 注入防护

- **描述：** 过滤用户输入中尝试覆盖系统指令的模式。
- **验收标准：**
  1. 检测并过滤常见的 prompt injection 模式（如 "Ignore all previous instructions"）
  2. AI 响应中的代码块和 HTML 转义处理
  3. 基于 callerRole 动态限制 AI Agent 可用的工具列表（学生不可调用高危工具）
- **影响文件：** `server.ts` AI 聊天路由

## Phase 18: server.ts 模块化拆分

### ARCH-SPLIT-01: 路由模块提取

- **描述：** 将 `server.ts` 中的 40+ 路由按业务领域拆分到独立模块。
- **验收标准：**
  1. 至少 12 个路由模块文件，每个 < 400 行
  2. `server.ts` 主文件 < 200 行
  3. 所有现有 `/api/*` 端点 URL 和响应格式不变
  4. Socket.IO 事件处理迁移到对应路由模块
- **目标结构：** 见 ROADMAP.md Phase 18 节

### ARCH-SPLIT-02: 中间件抽象

- **描述：** 将重复的 auth 检查、参数校验、权限判断抽取为 Express 中间件。
- **验收标准：**
  1. `requireAuth(...)` 中间件替代 30+ 处内联 session cookie 解析
  2. `validate(schema)` 中间件替代手动参数校验
  3. `requireCapability(cap)` 中间件替代内联 capability check

### ARCH-SPLIT-03: 工具函数独立

- **描述：** 将 `server.ts` 中的大型辅助函数提取到独立工具模块。
- **验收标准：**
  1. `extractScoreCommentCompletion` → `server/utils/score-parser.ts`
  2. `injectLmsSdk`、`hasScoreDisplay`、`hasDataSubmission` → `server/utils/courseware.ts`
  3. `bridge.js` SDK 模板 → `server/services/bridge-sdk.ts`
  4. 文件上传校验函数 → `server/utils/file-upload.ts`

### ARCH-SPLIT-04: AI 服务抽象

- **描述：** 将 Gemini 和 OpenAI 的 AI 调用逻辑提取为独立服务模块。
- **验收标准：**
  1. `runGeminiAgentChat` → `server/services/agent-gemini.ts`
  2. `runOpenAIAgentChat` → `server/services/agent-openai.ts`
  3. 统一入口函数 `runAgentChat(provider, params)`
  4. 工具调用循环提取为独立函数

### ARCH-SPLIT-05: 回归测试覆盖

- **描述：** 确保拆分不引入回归。
- **验收标准：**
  1. 关键 API 端点有集成测试：auth（登录/登出）、lesson CRUD、agent chat、class/student CRUD
  2. 拆分前后相同请求返回相同响应
  3. Socket.IO 事件发布时序不变

## Phase 20: 数据库迁移系统与备份

### DB-MIG-01: 迁移框架搭建

- **描述：** 建立版本化数据库迁移系统，替代 try/catch ALTER TABLE。
- **验收标准：**
  1. `migrations/` 目录，SQL 文件命名 `NNN_description.sql`
  2. 每个迁移文件包含 `-- UP` 和 `-- DOWN` 两部分
  3. `_migrations` 元表：name TEXT, applied_at INTEGER, checksum TEXT
  4. 启动时自动应用未执行的迁移（按序号）
  5. 迁移执行失败时阻止服务启动

### DB-MIG-02: 现有 Schema 整合

- **描述：** 将所有分散的 DDL 语句整合到迁移文件。
- **验收标准：**
  1. `000_initial_schema.sql` — 初始 30+ 张表
  2. `001_add_users_status.sql` 等所有 ALTER TABLE → 迁移文件
  3. `server.ts` 中的 `student_rollcalls` → 迁移文件
  4. `db/index.ts` 仅负责初始化连接 + 运行迁移

### DB-MIG-03: 数据生命周期管理

- **描述：** 为 events、sessions、pending_commands 增加自动清理。
- **验收标准：**
  1. events 表保留 90 天（通过 ProcessManager 定时任务）
  2. client_sessions 过期自动清理
  3. pending_commands 超 30 天自动清理
  4. 清理频率：每天 3:00 AM

### DB-MIG-04: 自动备份

- **描述：** 实现数据库定期自动备份。
- **验收标准：**
  1. 每天备份一次（`VACUUM INTO`）
  2. 保留最近 7 天备份
  3. 备份路径通过 `BACKUP_DIR` 配置
  4. 备份失败记录 ERROR 日志（不中断服务）

## Phase 21: 可观测性基础设施

### OBS-LOG-01: 日志框架引入

- **描述：** 用 pino 替代 console.log/error，实现结构化日志。
- **验收标准：**
  1. 引入 pino，配置日志级别（环境变量 `LOG_LEVEL`）
  2. 开发环境使用 `pino-pretty`，生产环境 JSON 输出
  3. 所有 `console.log/error` 替换为 `logger.info/error`

### OBS-LOG-02: 日志上下文标准化

- **描述：** 确保每条日志携带足够的上下文信息用于诊断。
- **验收标准：**
  1. 日志自动携带：timestamp、level、component、requestId
  2. HTTP 请求日志记录：method、url、status、duration、(userId)
  3. CommandBus 日志记录：commandType、actorId、duration

### OBS-LOG-03: 审计日志强化

- **描述：** 增强 events 表，提供查询接口。
- **验收标准：**
  1. events 表增加 actor_ip、user_agent 列
  2. `GET /api/admin/audit-log` 查询接口（时间范围、类型过滤、分页）

### OBS-HEALTH-01: 健康检查端点

- **描述：** 添加标准的健康检查端点。
- **验收标准：**
  1. `GET /health` — 基础存活检查
  2. `GET /health/ready` — 深度就绪检查（DB + Worker）
  3. `GET /metrics` — 基础性能指标

### OBS-SHUTDOWN-01: 优雅关闭

- **描述：** 实现 SIGTERM/SIGINT 的优雅处理。
- **验收标准：**
  1. 注册信号处理器
  2. 关闭顺序：HTTP → Socket.IO → Worker → DB
  3. 30s 超时后强制退出
  4. 关闭过程记录日志

### OBS-ERROR-01: 错误追踪

- **描述：** 统一错误处理，增强错误上下文。
- **验收标准：**
  1. 全局错误处理中间件
  2. 关键错误触发 ERROR 日志（含堆栈）
  3. 预留 Sentry 集成接口

## 不在范围内 (Out of Scope)

| 功能 | 原因 |
|------|------|
| 插件市场/商店 | 需要 CDN、审计、付费等基础设施，独立里程碑 |
| 第三方插件 iframe 沙箱 | 当前仅限内部第一方插件，独立安全里程碑 |
