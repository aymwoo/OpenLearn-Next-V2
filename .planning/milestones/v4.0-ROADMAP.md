# Roadmap: OpenLearnV2 v4.0 — 质量基础与生产就绪

**里程碑目标：** 将 OpenLearnV2 从"功能完整"提升至"生产就绪"，系统性地解决安全漏洞、架构债务、可观测性缺失三大类问题，建立持续集成与代码规范基础。

**前置条件：** v3.0 作业提交与学生互评插件（已完成）

---

## Phases 总览

| Phase | 名称 | 主题 | 优先级 |
|-------|------|------|--------|
| 17 | 安全加固 | 身份认证、数据保护、攻击面收缩 | 🔴 高 |
| 18 | server.ts 模块化拆分 | 路由提取、中间件抽象、工具函数独立 | 🔴 高 |

| 22 | CI/CD 与代码规范 | ESLint/Prettier、GitHub Actions、功能补全 | 🟢 常规 |

## Phase 17: 安全加固

**目标：** 修复所有已知的中等及以上严重度安全漏洞，使系统达到基本安全基线。

**需求：** SEC-AUTH-01 ~ SEC-AUTH-05, SEC-DATA-01 ~ SEC-DATA-03, SEC-NET-01 ~ SEC-NET-04

### 17.1 身份认证安全

#### SEC-AUTH-01: 学生密码哈希化
- **当前问题：** `students.password` 列存储明文密码，默认密码 `123456`，完全无保护
- **位置：** `server.ts`（学生创建/更新路由）、`packages/core/db/index.ts`（students 表定义）
- **验收标准：**
  - 学生密码使用 bcrypt/argon2 哈希存储（cost factor ≥ 10）
  - 移除默认密码 `123456` 的硬编码
  - 首次创建学生时生成随机临时密码，通过教师渠道分发
  - 学生登录验证改用哈希比对
  - 提供密码强度校验（最小长度、字符组合要求）

#### SEC-AUTH-02: 教师/管理员密码升级
- **当前问题：** 使用无盐单轮 SHA-256（`crypto.createHash('sha256').update(pwd).digest('hex')`），易受彩虹表攻击
- **位置：** `packages/core/db/index.ts:472`（种子用户）、`server.ts`（登录验证）
- **验收标准：**
  - 统一迁移到 bcrypt（与 SEC-AUTH-01 使用相同方案）
  - 实现旧哈希自动升级：登录时用旧方式验证通过后，自动重新哈希为新格式
  - 提供 `npm run migrate-passwords` 脚本批量升级现有用户密码

#### SEC-AUTH-03: Session 安全强化
- **当前问题：** Token 无过期机制（Max-Age=1年）、无 Secure 标志、无用户关联、无服务端清理
- **位置：** `server.ts` Set-Cookie 逻辑、登录路由
- **验收标准：**
  - 生产环境设置 `Secure` 标志
  - 实现 session 过期：空闲 24 小时自动失效，绝对有效期 7 天
  - `client_sessions` 表增加 `user_id` 外键关联
  - 每次验证时检查过期，自动清理过期 session
  - 用户修改密码后所有 session 失效

#### SEC-AUTH-04: 登录频率限制
- **当前问题：** `/api/auth/login` 无任何频率限制，可暴力破解
- **位置：** `server.ts` 登录路由
- **验收标准：**
  - 使用 `express-rate-limit` 对登录端点限流（5次/IP/分钟）
  - 连续失败后实施递增延迟（指数退避）
  - 记录并告警异常高频登录尝试

#### SEC-AUTH-05: 学生自助密码修改
- **当前需求：** 学生无法自行修改密码
- **验收标准：**
  - 提供 `POST /api/auth/change-password` 端点（需验证旧密码）
  - 前端学生面板增加密码修改 UI
  - 密码修改后使旧 session 失效

### 17.2 数据安全

#### SEC-DATA-01: AI Provider API Key 加密存储
- **当前问题：** API Key 明文存储在 `ai_providers.api_key`，且在 `/api/ai-providers` 响应中完整返回
- **位置：** `packages/core/db/index.ts`（表定义）、`server.ts`（API路由、AI调用）
- **验收标准：**
  - 使用 AES-256-GCM 加密存储 API Key，密钥通过环境变量 `ENCRYPTION_KEY` 注入
  - `/api/ai-providers` 返回时 mask API Key（显示前4后4字符，中间 `****`）
  - 解密仅在服务端发起 AI 调用时进行，不向外暴露

#### SEC-DATA-02: 学生数据隐私合规
- **当前问题：** 无数据导出、无数据删除、无 GDPR/CCPA 合规措施
- **验收标准：**
  - `GET /api/students/:id/export` — 导出学生所有个人数据（JSON 格式）
  - `DELETE /api/students/:id/gdpr-delete` — 完全删除学生数据（需管理员审批）
  - 数据保留策略文档化

#### SEC-DATA-03: 文件上传安全校验
- **当前问题：** 上传仅检查扩展名，无 magic bytes 校验，无 zip bomb 防护
- **位置：** `server.ts` 上传路由
- **验收标准：**
  - 所有文件上传校验 magic bytes（文件头魔数），拒绝扩展名与内容不匹配的文件
  - ZIP 上传增加解压后总大小限制（zip bomb 防护）
  - 上传文件大小按端点区分限制（课件 50MB，插件 10MB，作业 20MB）
  - 可执行文件类型（.exe, .sh, .bat 等）拒绝上传

### 17.3 网络安全

#### SEC-NET-01: CORS 白名单化
- **当前问题：** Socket.IO `cors: { origin: '*' }` 允许任意来源
- **位置：** `server.ts` Socket.IO 初始化
- **验收标准：**
  - 生产环境限制为实际前端域名
  - 开发环境允许 `localhost:5173` / `localhost:9000`
  - 通过环境变量 `ALLOWED_ORIGINS` 配置

#### SEC-NET-02: 安全 HTTP 头
- **当前问题：** 未使用 helmet，缺少基本安全头
- **验收标准：**
  - 集成 `helmet` 中间件
  - 配置 Content-Security-Policy（限制 script-src、防止 XSS）
  - 配置 Strict-Transport-Security（HSTS）
  - 配置 X-Content-Type-Options: nosniff

#### SEC-NET-03: 输入验证与消毒
- **当前问题：** 所有路由手动校验，无统一输入消毒
- **验收标准：**
  - 引入 zod 作为请求体验证 schema（与插件 manifest 校验保持一致）
  - AI Agent 聊天消息进行 XSS 消毒（防止 prompt 注入反射）
  - 所有数据库写入参数使用参数化查询（已满足，需审计确认）

#### SEC-NET-04: AI Prompt 注入防护
- **当前问题：** 用户消息和附件内容直接注入 AI prompt
- **验收标准：**
  - 用户输入进行敏感模式过滤（如系统指令覆盖尝试）
  - AI 输出进行 HTML 标签转义后再渲染到前端
  - 限制 AI Agent 可调用的工具范围（基于 callerRole 动态过滤）

## Phase 18: server.ts 模块化拆分

**目标：** 将 5000+ 行的单体 `server.ts` 拆分为按领域组织的路由模块，提取中间件和工具函数，每个模块可独立理解和测试。

**需求：** ARCH-SPLIT-01 ~ ARCH-SPLIT-05

### 18.1 路由模块提取

按业务领域拆分，每个模块自包含路由注册逻辑：

```
server/
├── server.ts              # 主入口：Express 配置、中间件、端口监听（<200行）
├── middleware/
│   ├── auth.ts            # session 验证中间件
│   ├── rate-limit.ts      # 频率限制中间件
│   └── validation.ts      # zod 请求体验证中间件
├── routes/
│   ├── auth.ts            # 登录/登出/session 管理
│   ├── lessons.ts         # 课程 CRUD
│   ├── whiteboard.ts      # 白板操作
│   ├── courseware.ts      # 课件上传/运行时
│   ├── classes.ts         # 班级/学生管理
│   ├── assignments.ts     # 作业/互评
│   ├── ai-chat.ts         # AI Agent 聊天
│   ├── resources.ts       # 系统资源
│   ├── plugins.ts         # 插件管理 API
│   ├── exams.ts           # 考试/成绩
│   ├── schedules.ts       # 排课/考勤
│   ├── mfe.ts             # 微前端远程入口
│   └── admin.ts           # 管理面板 API
├── services/
│   ├── agent-gemini.ts    # Gemini AI 调用逻辑
│   ├── agent-openai.ts    # OpenAI 兼容 AI 调用逻辑
│   ├── bridge-sdk.ts      # LMS SDK 注入逻辑
│   └── ocr.ts             # OCR 处理逻辑
└── utils/
    ├── score-parser.ts    # extractScoreCommentCompletion 等
    ├── courseware.ts      # injectLmsSdk, hasScoreDisplay 等
    └── file-upload.ts     # magic bytes 校验、文件保存
```

#### ARCH-SPLIT-01: 路由模块拆分
- **验收标准：**
  - 每个路由文件导出 `registerRoutes(app: Express)` 函数
  - `server.ts` 主文件缩减至 200 行以下
  - 所有现有 API 端点行为不变（API 兼容性测试通过）
  - Socket.IO 事件处理拆分到对应路由模块

#### ARCH-SPLIT-02: 中间件抽象
- **验收标准：**
  - 提取 `requireAuth(roles?: string[])` 中间件替代内联 session 检查
  - 提取 `validateBody(schema: ZodSchema)` 中间件
  - 提取 `requireCapability(cap: string)` 中间件
  - 所有路由使用中间件组合替代手动检查

#### ARCH-SPLIT-03: 工具函数独立
- **验收标准：**
  - `extractScoreCommentCompletion` → `server/utils/score-parser.ts`
  - `injectLmsSdk` → `server/utils/courseware.ts`
  - `bridge.js` SDK 代码 → `server/services/bridge-sdk.ts`
  - magic bytes 校验 → `server/utils/file-upload.ts`

#### ARCH-SPLIT-04: AI 服务抽象
- **验收标准：**
  - `runGeminiAgentChat` → `server/services/agent-gemini.ts`
  - `runOpenAIAgentChat` → `server/services/agent-openai.ts`
  - 统一 AI 调用接口：`runAgentChat(provider, params)`
  - 新增 AI 提供商只需实现接口，无需修改路由

#### ARCH-SPLIT-05: 回归测试覆盖
- **验收标准：**
  - 关键 API 端点有集成测试（auth、lesson CRUD、agent chat）
  - 拆分前后的 API 响应格式一致
  - Socket.IO 事件触发时序不变

## Phase 19: App.tsx 状态抽取与组件化

**目标：** 将 11159 行的 `App.tsx` 中的状态管理和业务逻辑抽取到 Zustand stores 和自定义 Hooks，按视图拆分为独立页面组件。

**需求：** FE-REFACTOR-01 ~ FE-REFACTOR-04

### 19.1 Zustand 状态迁移

#### FE-REFACTOR-01: 核心状态 Store 化
- **当前问题：** 160+ `useState` 调用全部在 App.tsx 中，整个组件树在任意状态变更时重渲染
- **验收标准：**
  - 创建以下 Zustand stores：
    - `useAuthStore` — 用户登录状态、角色、session
    - `useLessonStore` — 当前课程、课程列表、环节时间线
    - `useClassStore` — 班级列表、学生花名册
    - `useWhiteboardStore` — 白板元素、当前页、选中工具
    - `useUIStore` — Tab 切换、面板展开/折叠、语言偏好
    - `useAgentStore` — AI 对话历史、当前对话
  - App.tsx 的 useState 数量减少 60%+
  - 组件仅在关联的 store slice 变更时重渲染

#### FE-REFACTOR-02: 自定义 Hook 提取
- **验收标准：**
  - `useSocket` — Socket.IO 连接管理和事件订阅
  - `useAgentChat` — AI 对话逻辑（发送、流式接收、工具调用展示）
  - `useCourseware` — 课件加载、提交、成绩解析
  - `useAttendance` — 考勤数据获取和更新
  - `useDataFetching` — 通用数据获取（loading/error/data 三态）
  - 每个 Hook 独立可测试

#### FE-REFACTOR-03: 页面组件拆分
- **验收标准：**
  - 教师视图拆分为独立页面组件：
    - `TeacherClassroomPage` — 上课面板（白板+课件+环节控制）
    - `TeacherLessonPage` — 课程管理
    - `TeacherAnalyticsPage` — 成绩分析图表
    - `TeacherManagementPage` — 班级/学生管理
    - `TeacherPluginPage` — 插件中心
    - `TeacherAdminPage` — 系统管理
  - 学生视图拆分为：
    - `StudentLessonPage` — 学习面板
    - `StudentAssignmentPage` — 作业/互评
    - `StudentProfilePage` — 个人中心

#### FE-REFACTOR-04: 性能优化
- **验收标准：**
  - 使用 `React.memo` 包裹纯展示组件
  - 使用 `useMemo`/`useCallback` 避免不必要的重新计算
  - 大列表使用虚拟滚动
  - 图表组件懒加载（`React.lazy` + `Suspense`）

## Phase 20: 数据库迁移系统与备份

**目标：** 替代当前的 try/catch ALTER TABLE 模式，建立版本化、可追溯、可回滚的数据库迁移系统，并实现自动备份策略。

**需求：** DB-MIG-01 ~ DB-MIG-04

### 20.1 迁移系统

#### DB-MIG-01: 迁移框架搭建
- **当前问题：** Schema 变更靠 `ALTER TABLE ... try/catch`，无法追溯和回滚
- **验收标准：**
  - 引入轻量级迁移库（如 `better-sqlite3` 内置迁移支持或自建）
  - `migrations/` 目录，命名规范：`001_add_users_status.sql`、`002_add_plugin_zip.sql`
  - `_migrations` 元表追踪已应用的迁移（name, applied_at, checksum）
  - 启动时自动应用未执行的迁移（按序号顺序）
  - 每个迁移文件包含 `-- UP` 和 `-- DOWN` 两部分

#### DB-MIG-02: 现有 Schema 整合
- **验收标准：**
  - 将 `db/index.ts` 中所有 `CREATE TABLE IF NOT EXISTS` 转换为初始化迁移 `000_initial_schema.sql`
  - 将 `server.ts` 中的增量 `CREATE TABLE`（如 `student_rollcalls`）整合到迁移文件
  - 将所有 `ALTER TABLE ADD COLUMN` try/catch 块转换为对应的迁移文件
  - 迁移序号基于时间线正确排序

#### DB-MIG-03: 数据生命周期管理
- **当前问题：** `events` 审计表无限增长
- **验收标准：**
  - 实现 events 表自动清理：保留最近 90 天数据
  - `client_sessions` 过期自动清理
  - `pending_commands` 超过 30 天的自动清理
  - 清理策略通过 ProcessManager 定时任务执行

#### DB-MIG-04: 自动备份
- **验收标准：**
  - 使用 `VACUUM INTO` 或文件复制实现数据库备份
  - 备份频率：每天一次，保留最近 7 天
  - 备份文件命名：`educational_os_backup_YYYY-MM-DD.db`
  - 通过 ProcessManager 定时任务执行
  - 备份文件存储路径通过环境变量 `BACKUP_DIR` 配置

## Phase 21: 可观测性基础设施

**目标：** 建立结构化日志、错误追踪、健康检查和优雅关闭机制，使系统在运行时可被监控和诊断。

**需求：** OBS-LOG-01 ~ OBS-LOG-03, OBS-HEALTH-01, OBS-SHUTDOWN-01, OBS-ERROR-01

### 21.1 结构化日志

#### OBS-LOG-01: 日志框架引入
- **当前问题：** 仅使用 `console.log/error`，无日志级别，无结构化格式，无持久化
- **验收标准：**
  - 引入 `pino` 作为结构化日志库
  - 定义日志级别：trace / debug / info / warn / error / fatal
  - 通过环境变量 `LOG_LEVEL` 控制日志级别
  - 开发环境输出美化格式（pino-pretty），生产环境输出 JSON
  - 所有 `console.log` 调用替换为 logger 调用

#### OBS-LOG-02: 日志上下文标准化
- **验收标准：**
  - 每条日志自动携带：timestamp、level、component（如 `[CommandBus]`）、requestId
  - HTTP 请求日志自动记录：method、url、status、duration、userId
  - CommandBus 执行日志记录：commandType、actorId、duration、success/failure
  - AI Agent 调用日志记录：provider、model、token 消耗、延迟

#### OBS-LOG-03: 审计日志强化
- **验收标准：**
  - 敏感操作必须同时写 events 表和结构化日志（双写）
  - events 表增加 `actor_ip` 和 `user_agent` 字段
  - 提供 `GET /api/admin/audit-log` 查询接口（支持时间范围和操作类型过滤）

### 21.2 健康检查与监控

#### OBS-HEALTH-01: 健康检查端点
- **验收标准：**
  - `GET /health` — 返回 200 + `{ status: "ok", uptime, version }`
  - `GET /health/ready` — 深度检查：DB 连接、AI Provider 可用性、Worker 状态
  - `GET /metrics` — 基础指标：请求数、响应时间分布、活跃连接数、内存使用
  - 数据库连接池状态检查

### 21.3 优雅关闭

#### OBS-SHUTDOWN-01: 优雅关闭
- **当前问题：** `startServer()` 无 SIGTERM/SIGINT 处理，直接杀进程可能导致数据丢失
- **验收标准：**
  - 注册 SIGTERM/SIGINT 信号处理器
  - 关闭顺序：停止接受新请求 → 等待进行中请求完成（30s 超时）→ 关闭 Socket.IO 连接 → 停止 Worker → 关闭数据库连接
  - 超时后强制退出
  - 关闭过程中记录 INFO 级别日志

### 21.4 错误追踪

#### OBS-ERROR-01: 错误上下文增强
- **验收标准：**
  - 所有 `try/catch` 中的错误记录包含堆栈信息和上下文参数
  - 定义项目级错误类层次（已有 Worker 端的，扩展到全局）
  - 关键错误（插件激活失败、数据库写入失败、AI 调用失败）触发 ERROR 级别日志
  - 预留 Sentry/错误追踪集成接口

## Phase 22: CI/CD 与代码规范 + 功能补全

**目标：** 建立持续集成流水线、统一代码风格、补全学生自助服务等缺失功能。

**需求：** CI-CD-01 ~ CI-CD-03, CODE-STD-01 ~ CODE-STD-02, FEAT-CMP-01 ~ FEAT-CMP-03

### 22.1 CI/CD

#### CI-CD-01: GitHub Actions 流水线
- **验收标准：**
  - PR 触发：`tsc --noEmit` 类型检查 + `vitest` 全量测试
  - 测试报告作为 PR comment 展示
  - main 分支 push 触发：构建 + 测试 + 生成构建产物
  - 定时任务：每周运行全部测试（防止依赖退化）

#### CI-CD-02: 依赖安全扫描
- **验收标准：**
  - `npm audit` 集成到 CI
  - Dependabot 配置（每周检查依赖更新）
  - 关键依赖锁定版本，次要依赖允许 semver 自动更新

#### CI-CD-03: Docker 化
- **验收标准：**
  - 编写 `Dockerfile`（多阶段构建：build + production）
  - `docker-compose.yml`（app + volume 持久化 DB）
  - `.dockerignore` 排除 node_modules、build artifacts

### 22.2 代码规范

#### CODE-STD-01: ESLint + Prettier 配置

#### CODE-STD-02: 代码风格统一
- **验收标准：**
  - 选择并统一缩进风格（推荐 2-space）
  - 统一文件命名：组件 PascalCase、工具 camelCase、常量 UPPER_SNAKE_CASE
  - 移除 `any` 类型，替换为具体类型或 `unknown`（渐进式，新增代码强制）
  - 启用 `tsconfig.json` strict mode（渐进式）
