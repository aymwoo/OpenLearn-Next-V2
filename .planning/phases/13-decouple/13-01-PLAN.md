# Phase 13 Plan: 业务模块解耦与样式沙箱化

<objective>
本规划旨在将 Host Shell 中的大单体 `App.tsx` 进行拆分，彻底解耦“协同白板”与“课件查看器”视图。
通过将其分别抽离为独立的微前端子应用项目（`packages/mfe-whiteboard` 与 `packages/mfe-courseware`），实现：
1. 彻底的前后端业务代码与依赖包解耦（独立发版、独立打包）。
2. 基于 Tailwind CSS v4 前缀（`wb:` / `cw:`）、禁用 Preflight 以及 CSS Modules 的轻量、低副作用 CSS 隔离与样式沙箱。
3. 宿主端基于 SQLite 数据库 `mfe_remotes` 的动态配置与种子自动注入。
4. 宿主端 UI 彻底下线大单体静态导入，改为 `MfeLoader` 异步动态渲染并搭配友好的停用/故障降级占位符（Fail-Safe 容灾）。
5. 完备的单元/集成自动化测试覆盖。
</objective>

<threat_model>
### Threat T-13-01: CSS Selector Pollution & Layout Hijack (CSS 样式污染与布局篡改)
*   **Description**: 子应用的默认样式（特别是 Tailwind CSS 内置的全局 Preflight 重置样式，如 `html`, `body`, `*` 的 margin/padding 重置）被加载至 Host 宿主后，会覆盖宿主的已有默认样式，导致宿主布局被篡改或破坏。
*   **Mitigation**:
    1. 在子应用的 `index.css` 中仅导入 `@layer theme, utilities` 并屏蔽 preflight 基础重置样式（`preflight: false`）。
    2. 子应用中通过 `@import "tailwindcss/utilities" prefix(wb)` 为所有 Tailwind 工具类自动加上 namespace 前缀（白板使用 `wb:` 前缀，课件使用 `cw:` 前缀）。
    3. 子应用内部编写的非 Tailwind 自定义样式一律强制使用 CSS Modules (`*.module.css`) 进行哈希混淆重命名。

### Threat T-13-02: Malicious MFE Entry Point Modification (恶意微应用入口修改 / 远程代码注入)
*   **Description**: 攻击者通过注入恶意 MFE entry 地址以试图在宿主浏览器上下文中运行任意 JS 脚本。
*   **Mitigation**:
    1. 动态 Remote 的 Entry URL 统一由系统管理员在服务器侧 SQLite 数据库表 `mfe_remotes` 中配置，禁止任何客户端及用户参数传入。
    2. 在 `packages/core/db/index.ts` 中加入安全的、写死的本地白名单种子配置，防止宿主端加载非授权来源脚本。
    3. 在 `MfeLoader` 外层使用 Error Boundary，当拉取非白名单/非法 entry 发生网络拦截或资源异常时，彻底阻断请求并友好降级显示，规避网络泛滥和潜在注入攻击。

### Threat T-13-03: Client State Exposure & Whitelisted DI Resolving (客户端状态泄露与非授信依赖解析)
*   **Description**: 微应用被非法注入后试图任意解析宿主内核的敏感微服务或窃取非授权的全局客户端状态。
*   **Mitigation**:
    1. DI 容器中采用白名单机制：子应用只能通过注入的 `mfeContext?.serviceRegistry` 解析预先注册的桥接服务标识（例如 `@openlearn/frontend:ISocketService`）。
    2. Zustand Store 采取单向的细粒度属性读取订阅，白板的笔画粗细、笔画颜色等大量局部状态保留在子项目本地的 `useState` 中，避免污染或泄漏到宿主全局状态树。
</threat_model>
