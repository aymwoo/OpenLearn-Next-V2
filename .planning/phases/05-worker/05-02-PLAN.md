<objective>
Transparent RPC layer with CapabilityGuard enforcement for Worker-to-main-thread service calls.

Purpose: Enable Worker-side plugin code to call `ctx.services.commandBus.execute(cmd)` transparently while the actual call happens on the main thread through a JavaScript Proxy. ServiceHost on the main thread resolves services, enforces capability checks (per plugin manifest), serializes errors, and returns results. The Worker-side plugin code should not be aware it is calling across a thread boundary.

Output: `service-proxy.ts` (Worker-side Proxy traps), `service-host.ts` (main thread RPC handler with CapGuard), `__tests__/service-proxy.test.ts`, `__tests__/service-host.test.ts`
</objective>

    **A) Create `service-proxy.ts`** implementing the Worker-side transparent RPC Proxy. This file runs INSIDE the Worker thread.

    Export three functions:

    1. `createMethodProxy(transport: IWorkerTransport, token: string, pendingCalls: Map<string, PendingCall>, timeoutMs?: number): Record<string, Function>`
       - Returns `new Proxy({} as Record<string, Function>, { get(target, method) { ... } })`.
       - The `get` trap returns a function `(...args: unknown[]) => Promise<unknown>`.
       - When called: generates `invokeId` via `crypto.randomUUID()`, creates Promise with resolve/reject stored in `pendingCalls` Map, calls `transport.postMessage({ type: 'invoke', invokeId, token, method, args })`.
       - Optional timeout: if `timeoutMs` is set, `Promise.race` against a timeout that rejects with `WorkerTimeoutError(timeoutMs)`.
       - Default `timeoutMs`: 30000 (30 seconds) — configurable via parameter.
       - The returned Promise resolves when a matching `ResultMessage` arrives, rejects when `ErrorMessage` arrives.

    2. `createServicesProxy(transport: IWorkerTransport, serviceTokens: string[]): { services: Record<string, Record<string, Function>>; pendingCalls: Map<string, PendingCall>; dispose: () => void }`
       - Creates a shared `pendingCalls` Map.
       - Registers `transport.onMessage` handler that checks message type:
         - `'result'` → finds pending call by invokeId, resolves pending promise, deletes from Map.
         - `'error'` → finds pending call by invokeId, creates Error from `{ message, code, stack }`, rejects pending promise, deletes from Map.
         - `'event'` → currently no-op (reserved for Plan 4 EventBusProxy).
       - The `dispose` function clears the onMessage handler registration and rejects all pending calls with `WorkerTransportError('Transport disposed')`.
       - Builds an object `services = {}`, then for each `token` string: `services[token] = createMethodProxy(transport, token, pendingCalls)`.
       - Freezes the services object with `Object.freeze(services)` to prevent tampering (follows context-builder.ts pattern from PATTERNS.md line 169).

    Use `crypto.randomUUID()` for invokeId generation (built into Node.js v24, no dependency needed).

    Do NOT export a class — the module exports functional factories, matching the pattern from `context-builder.ts` (exports `buildContext`, `createSafeFunction`).

    **A) Create `service-host.ts`** implementing the main-thread RPC handler.

    Export a `ServiceHost` class with:

    1. **Constructor**: `(serviceRegistry: ServiceRegistry, capabilityGuard: CapabilityGuard, pluginActorId: string, manifestCapabilities: string[])`
       - `serviceRegistry`: the DI container for resolving service instances.
       - `capabilityGuard`: the capability guard instance (NOT the async ICapabilityService wrapper — use CapabilityGuard directly since we're on the main thread).
       - `pluginActorId`: fixed at construction (e.g., `'plugin:ext-quiz-generator'`). All invokes from this Worker are attributed to this actor.
       - `manifestCapabilities`: array of capability strings from manifest.capabilitiesProposed. Used for checking.

    2. **`handleMessage(msg: unknown, transport: IWorkerTransport): Promise<void>`**
       - Dispatches based on msg.type:
         - `'invoke'` → routes to `handleInvoke(msg, transport)`.
         - `'subscribe'` → currently no-op with `console.warn` (Plan 4 will implement).
         - `'unsubscribe'` → currently no-op (Plan 4).
         - `'activated'` → no-op (recognized acknowledgement).
         - `'deactivated'` → no-op (acknowledgement).
         - unknown type → no-op (defensive, don't throw).
       - Wrap in try/catch — never let handler exception crash the message loop.

    3. **`handleInvoke(msg: InvokeMessage, transport: IWorkerTransport): Promise<void>`** — the core RPC handler:
       a. **Resolve service by token string**: Call `serviceRegistry.resolveByName(msg.token)`.
          If `resolveByName` does NOT exist on ServiceRegistry (it was added as a new method per RESEARCH.md open question #1), call the existing `resolve()` by constructing a temporary Token. But the ideal path: add `resolveByName(name: string)` to ServiceRegistry that does `this.registry.get(name)?.instance`. The executor should check if resolveByName exists; if not, implement it as a 5-line addition to `../di/service-registry.ts`.
       b. **Extract the service instance**: from the resolved entry.
       c. **Get the method**: `const method = (service as any)[msg.method]`. If not a function, throw `WorkerActivateError` with message.
       d. **CapabilityGuard check**: For the specific service invocation, determine which capability is required:
          - The simplest approach: check `this.manifestCapabilities` is non-empty, and for ALL invokes from a Worker plugin, check the most specific capability we can derive.
          - A pragmatic rule: deny invoke if manifestCapabilities is empty AND the plugin is NOT an admin bypass. If manifestCapabilities is non-empty, check `this.capabilityGuard.check(this.pluginActorId, '*')` as a gating check. Full per-method capability mapping is deferred to a separate concern — for Phase 5, we block Workers with empty capabilities from making ANY service calls beyond a whitelist of read-only methods.
          - Implementation: `const allowed = this.capabilityGuard.check(this.pluginActorId, '__worker_invoke__')` where `__worker_invoke__` is a pseudo-capability that is automatically granted during `grant()` for all manifest-c claimed capabilities. Actually, simpler: use `this.manifestCapabilities.length === 0` as the gating check — if empty, only allow `storage.get` (read-only). This is documented as "Phase 5 pragmatic rule" in comments.
          - Actual enforcement: `if (this.manifestCapabilities.length === 0 && msg.method !== 'get') { throw new WorkerCapabilityError(this.pluginActorId, '__rpc__'); }`
       e. **Execute**: `const result = await method.apply(service, msg.args)`.
       f. **Return**: `transport.postMessage({ type: 'result', invokeId: msg.invokeId, value: result })`.
       g. **Error**: Catch any error, serialize as `{ type: 'error', invokeId: msg.invokeId, message: err.message, code: err.name, stack: err.stack }`. Cap stack to 4096 chars.
       h. **Structured clone safety**: The `result` value must be structured-clone-safe. If result contains functions/Symbols/WeakRefs, structured clone will throw DataCloneError. Document this constraint in a comment.

    Additional method:

    4. **`get actorId(): string`** — returns the pluginActorId. Useful for error messages and logging.
    5. **`setManifestCapabilities(caps: string[]): void`** — allows updating capabilities after construction (for dynamic capability grants).

    IMPORTANT: Do NOT add event subscription handling in this plan. The handleSubscribe/handleUnsubscribe methods in handleMessage should log a warning. Event forwarding is Plan 4.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker -> ServiceHost | Untrusted Worker sends 'invoke' messages. ServiceHost processes trusted = false. |
| ServiceHost -> ServiceRegistry | Trusted. ServiceHost runs on main thread with full system access. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Elevation of Privilege | service-host.ts handleInvoke | mitigate | CapabilityGuard.check(actorId) called BEFORE every service method execution. Worker actorId is fixed at construction and derived from manifest — Worker cannot forge actorId in invoke messages. |
| T-05-05 | Elevation of Privilege | service-proxy.ts Proxy traps | mitigate | The Proxy only serializes method+args. It cannot call arbitrary functions — only methods on the resolved service instance. Token string is fixed per proxy, not supplied by caller. |
| T-05-06 | Information Disclosure | service-host.ts error serialization | mitigate | Stack traces capped at 4096 chars. Error messages may contain path/implementation details — this is acceptable since the Worker belongs to the same deployment. |
| T-05-07 | Denial of Service | service-host.ts handleInvoke | accept | Malformed invoke messages are caught by the wrapper try/catch. However, a rapid-fire invoke flood is handled at the transport level (not ServiceHost). Monitoring/logging of abandoned pendingCalls (invokeIds never matched) can detect leaks. |
| T-05-08 | Tampering | service-host.ts resolveByName | mitigate | The token string is passed by the Worker proxy at construction time, not by user input. Token string is validated by the discriminator — only known tokens from the service list are passed. |
</threat_model>
