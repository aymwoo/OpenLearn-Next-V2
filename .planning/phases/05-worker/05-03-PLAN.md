## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Plugin boundary (inline vs worker) | Trusted built-in plugins run inline; untrusted third-party run in Worker. Mode detection is the gating decision. |
| WorkerRegistry -> Worker | WorkerRegistry tracks but does not trust Workers. Termination uses finally block for guaranteed cleanup. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | Denial of Service | worker-manager.ts Worker creation | mitigate | WorkerRegistry.activeCount capped at a configurable max (default 32). createWorker throws when limit reached. |
| T-05-10 | Elevation of Privilege | plugin-host/index.ts mode detection | mitigate | execution_mode is read from DB (source of truth), not from user input passed in the options parameter. DB value takes precedence if set. Options parameter is only honored at initial activation. |
| T-05-11 | Denial of Service | worker-manager.ts terminateWorker | mitigate | Worker termination in finally block guarantees cleanup. If Worker is unresponsive (deactivate timeout), forced `worker.terminate()` kills the thread. |
| T-05-12 | Tampering | plugin-host/index.ts activateWorker | mitigate | Worker constructor catches failures, rolls back plugin state to ERROR, terminates Worker if start failed. Single Worker failure does not cascade to other plugins. |
| T-05-13 | Spoofing | worker-manager.ts Worker registry | mitigate | WorkerRegistry maps pluginId to WorkerInstance. Cross-Worker identity spoofing is impossible because the transport channel is paired 1:1 with the Worker at creation time. |
| T-05-14 | Tampering | db/index.ts execution_mode | accept | The execution_mode column is set at install time and read-only thereafter. An attacker who can write to the DB can change the mode, but that requires DB access which is already root-level access. |
