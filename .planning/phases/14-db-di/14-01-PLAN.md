# Phase 14 Plan: 数据库结构设计与后端 DI 成绩对接服务

<objective>
本规划旨在设计并实现“上课作业提交与互评插件”的底层数据存取及成绩同步对接架构。
通过定义跨越前后端的 `ISemesterGradeService` 以及对 MFE 白名单的开放，实现以下目标：
1. idempotent 创建三张自定义插件数据库表：`plugin_submissions`、`plugin_peer_reviews`、`plugin_grades`。
2. 抽象及实现平时成绩同步服务 `ISemesterGradeService`，打通平时确认成绩同步到宿主原有 assignments 与 assignment_submissions 结构，自动核算进学期 assignment_score 平均分。
3. 创建独立的后端逻辑插件 `AssignmentEvalPlugin`，承载作业提交、学生互评、打分等级等核心 Command 处理及业务校验，避免代码污染宿主核心。
4. 完成完备的单元测试，保证数据库与 DI 通道健壮性。
</objective>

<threat_model>
- **Threat T-14-01: 恶意互评分数绕过限制 (Invalid Peer-Review Scores / Collusion)**
  - **Mitigation**: 后端 `assignment.peer_review` 处理器执行严格边界防御，限制评分必须在 0 到 100 的闭区间，且通过校验拦截自评行为（`reviewerId === studentId`）。
- **Threat T-14-02: 绕过白名单注入伪造成绩同步服务 (Malicious DI Resolve / Grade Forgery)**
  - **Mitigation**: 前端的 `MfeServiceRegistryProxy` 对该服务进行隔离，只保留 `saveSemesterGrade` 只读映射操作；后端在 `/api/grade-sync` 接口对发起成绩同步的 `actorId` 做严格角色验证，非 teacher 角色（或者是无 lesson:write 权限）发起的一律拒绝，防止学生利用开发者工具伪造最终平时分。
- **Threat T-14-03: 平时分伪造溢出与表注入 (SQL Injection / Integer Overflow)**
  - **Mitigation**: 所有成绩数据写入一律采用 SQL 参数化预编译语句（Prepared Statements），禁止拼接字符串；分数强制使用 `Math.round()` 或 `parseInt` 保证入库为合法数值。
</threat_model>
