    - registerOrReplace：调用 removeEdges 清理旧实例的所有依赖边（requires/dependents 双向清理），删除旧 registry 条目，再调用 register 插入新实例（per D-08, RESEARCH.md Pattern 3）

实现 packages/core/di/service-registry.ts — ServiceRegistry 类（per 上述 behavior 规格）：

内部数据结构：
- `private registry = new Map<string, ServiceEntry>()` — Token name → 服务实例 + 注册选项
- `private depGraph = new Map<string, DepEdge>()` — Token name → 依赖边（requires/dependents）

公开方法（全部 async 返回 Promise，per D-05）：
- `async register<T>(token: Token<T>, instance: T, options?: RegisterOptions): Promise<void>`
- `async resolve<T>(token: Token<T>): Promise<T>`
- `async unregister<T>(token: Token<T>): Promise<void>`
- `async registerOrReplace<T>(token: Token<T>, instance: T, options?: RegisterOptions): Promise<void>`
- `list(): Array<{ name: string; instance: unknown }>`（内省 API，同步即可，不需要 async 但保持接口风格一致可定为同步）
- `has<T>(token: Token<T>): boolean`
- `dependencies(tokenName: string): { requires: string[]; dependents: string[] } | undefined`

关键实现细节：
- requires 数组在存储时去重：`new Set(options.requires ?? [])`
- depGraph 中的 require 边存储为 Set<string>，不是数组（per Pitfall 2）
- register 方法中先检查重复：`if (this.registry.has(token.name)) throw new DuplicateRegistrationError(token.name)`
- register 方法中检查依赖：遍历 requires，跳过 optional 中的依赖；未注册的 required dep 收集到数组，全部检查后统一抛出 MissingDependencyError
- 依赖图更新：对每个 require，设置 `depGraph.get(req)!.dependents.add(token.name)`（反向边）
- unregister 方法中 `dependents.size > 0` 检查：如果存在依赖方，抛出 HasDependentError(token.name, Array.from(dependents))
- unregister 清理：遍历 requires 边，对被依赖方执行 `depGraph.get(req).dependents.delete(token.name)`（清理反向引用）
- topologicalOrder 私有方法：Kahn BFS 算法，返回拓扑排序的 name 数组，剩余未入列节点抛出 CircularDependencyError
- registerOrReplace 检查旧实例 → removeEdges → 删除 → register 新实例
- removeEdges 私有方法：遍历 requires 边清理反向 dependents，删除 depGraph 条目

导入规范：使用 .js 扩展名 ESM 导入
从 token.ts 导入 Token 类型
从 errors.ts 导入 DuplicateRegistrationError, MissingDependencyError, CircularDependencyError, HasDependentError
从 types.ts 导入 RegisterOptions, ServiceEntry, DepEdge

不要添加装饰器、reflect-metadata、或任何外部 DI 库依赖（per D-01）

    - 未注册解析：resolve 未注册的 token → 抛出 'No provider' 错误（SC-5）
    - 依赖注册（简单）：先注册依赖（如 serviceA），再注册依赖方（serviceB requires [serviceA]）→ 两者均可 resolve；顺序相反则抛出 MissingDependencyError（D-06）
    - 拓扑排序依赖解析（链式）：A → B → C 依赖链 → 按 A→B→C 注册，全部可 resolve（SC-3）
    - 循环依赖检测：A requires B, B requires A → 循环检测抛出 CircularDependencyError，错误信息包含 A 和 B 的 name（SC-4）
    - 间接循环依赖：A→B→C→A 三节点环 → 抛出 CircularDependencyError
    - registerOrReplace：注册 A，registerOrReplace 用新实例覆盖 A → resolve 返回新实例；depGraph 中旧边被清除
    - unregister 无依赖方：注册 token → unregister → resolve 抛出 'No provider'
    - unregister 有依赖方阻止：A depends on B → unregister(B) 抛出 HasDependentError，错误信息包含 A 的 name（D-09）
    - unregister 级联清理验证：注册 A depends on B → unregister(A) 成功 → B 的 dependents 不再包含 A；has(A) 返回 false
    - 内省 API list()：注册 2 个 token → list() 返回长度为 2 的数组，包含 name 和 instance
    - 内省 API has()：注册 → has(token) true；未注册 → has(token) false
    - 内省 API dependencies()：注册 A depends on B → dependencies(A_name) 返回 { requires: [B_name], dependents: [] }；dependencies(B_name) 返回 { requires: [], dependents: [A_name] }
    - optional 依赖不强制检查：注册时 requires 缺失抛错，但 optional 缺失不抛错
