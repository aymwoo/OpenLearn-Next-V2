安装 tsc-strict CLI 工具（per D-18 和 RESEARCH.md Common Pitfall 1）：
运行 `pnpm add -D tsc-strict`（typescript-strict-plugin 的配套 CLI 工具，在 CLI 编译时应用 strict 规则）

配置 tsconfig.json 以启用 typescript-strict-plugin（如果尚未配置）：
- 在 compilerOptions 中添加 plugins 配置：
  `"plugins": [{ "name": "typescript-strict-plugin" }]`
- 注意：tsconfig.json 当前没有 plugins 字段，需要新增
- 该插件使用 "opt-out" 模型（默认行为）：所有文件自动 strict，仅在需排除的文件中加 `// @ts-strict-ignore` 注释
- 不要在 tsconfig.json 中设置全局 `"strict": true`——现有代码不符合 strict 标准，会导致 massive 类型错误。文件级 strict 是本阶段的范围（per D-18：仅 Token 和 ServiceRegistry 源码文件使用 strict 模式）

在 package.json 中更新 lint 脚本：
- 修改 `"lint": "tsc --noEmit && tsc-strict"` — 先运行 tsc 基础类型检查，再运行 tsc-strict CLI 检查
- tsc-strict 默认会检查所有文件，但在 di/ 目录外的文件因有 `// @ts-strict-ignore` 注释会被排除
- 同时添加 `"test": "vitest run"` 脚本（如果 Plan 01 尚未添加）

验证配置：
- 运行 `npx tsc --noEmit` 确认基础类型检查通过（不应因 plugins 配置而引入新错误，plugins 字段仅影响 IDE 语言服务）
- 运行 `npx tsc-strict` 确认 strict 检查可执行（预期：对 packages/core/di/ 下的文件执行 strict 检查，其他文件因 ignore 注释跳过）
- 确保 di/ 目录下的源码文件（token.ts、service-registry.ts、errors.ts、types.ts）没有 `// @ts-strict-ignore` 注释——它们应该接受 strict 检查

如果 tsc-strict 报告 di/ 目录文件中存在 strict 类型错误（如未初始化的类属性 `_phantomService!: T` 的 definite assignment assertion `!` 不被 strict 接受）：
- 修复源码使其通过 strict 检查
- Token 类的 phantom 字段可以使用 `private _phantomService!: T`（definite assignment assertion）——这在 strict 模式下应该被允许
- 如果 strict 不允许 `!`，改用 `// @ts-expect-error` 注释或直接赋值 `undefined as unknown as T`

最终验证：`pnpm run lint`（即 `tsc --noEmit && tsc-strict`）通过
