# Plan: 07-02 — Atomic Hot Reload Strategy

**Objective:** 实现 PluginHost.reloadPlugin() 的完整原子热重载策略——先激活新版本，成功后停用旧版本并清理资源；失败则完整保留旧版本运行状态。

## Tasks

### Task 1: 实现完整的 reloadPlugin() 方法

**Files:**
- `packages/core/plugin-host/index.ts` — 重写 reloadPlugin() 方法

**Description:**

在 PluginHost 中实现完整的原子热重载（替换 Task 5 的骨架）：

```typescript
/**
 * 原子热重载插件。
 *
 * 策略（atomic new-before-old）：
 * 1. 提取新 manifest，验证 ID 一致性
 * 2. 构建新 PluginContext（buildContext）
 * 3. ESM 加载新源码 → 调用新 activate(ctx)
 * 4. [成功] 停用旧版本 → 清理旧资源 → 替换实例引用 → 更新 DB
 * 5. [失败] 保留旧版本运行，清理临时资源，抛出 HotReloadActivationError
 *
 * @param pluginId - 插件标识符
 * @param newSourceCode - 新版本源码
 * @throws HotReloadActivationError 新版本激活失败时
 */
async reloadPlugin(pluginId: string, newSourceCode: string): Promise<void>
```

详细实现步骤:
1. **状态检查**: 插件必须处于 ACTIVE 状态，否则抛出 IllegalStateTransitionError
2. **提取 manifest**: `extractManifest(newSourceCode)` — 验证 manifest.id 与旧 manifest.id 一致
3. **SemVer 检查**: `checkSemVerCompatibility(newManifest, pluginId, 'reload')`
4. **快照旧资源** (CRITICAL — 防止 disposeAll 误伤新资源):
   ```
   const oldDisposables = this.resourceTracker.snapshot(pluginId);
   // ResourceTracker 新增 snapshot(pluginId) 方法:
   //   返回当前 Disposable[] 的浅拷贝，用于精确清理
   //   实现: return [...(this.resources.get(pluginId) ?? [])];
   ```
5. **构建 Context**: `buildContext(newManifest)` — 获取新的 ctx（含 services + skipTokens）
   - ContextBuilder 在 activate 期间会将新 disposables 注册到 ResourceTracker 的同一 pluginId
   - 因为步骤 4 已快照旧 disposables，新 disposables 不会被误清理
6. **ESM 加载 + 激活**:
   ```
   const pluginModule = await this.esmLoader.load(newSourceCode);
   const newInstance = { manifest: newManifest, activate: pluginModule.activate, deactivate: pluginModule.deactivate };
   // 应用 beforeActivate/afterActivate 中间件管道（07-03 集成点）
   const middlewareCtx = { pluginId, manifest: newManifest, phase: 'beforeActivate', timestamp: Date.now() };
   const activatePipeline = compose(this.getMiddleware('beforeActivate'));
   await activatePipeline(middlewareCtx, async () => {
     await newInstance.activate(ctx);
     // afterActivate 中间件
     const afterCtx = { ...middlewareCtx, phase: 'afterActivate' };
     const afterPipeline = compose(this.getMiddleware('afterActivate'));
     await afterPipeline(afterCtx, async () => {});
   });
   ```
7. **激活成功 → 清理旧版本**:
   ```
   const oldInstance = this.pluginInstances.get(pluginId);
   // 在 deactivate 前应用 beforeDeactivate 中间件
   const beforeDeactCtx = { ...middlewareCtx, phase: 'beforeDeactivate' };
   const beforeDeactPipeline = compose(this.getMiddleware('beforeDeactivate'));
   await beforeDeactPipeline(beforeDeactCtx, async () => {
     // 调用旧 deactivate
     if (oldInstance?.deactivate) {
       await withTimeout(oldInstance.deactivate(), 5000);
     }
   });
   // 精确清理旧资源（仅快照中的 disposables，不碰新注册的）
   for (const d of (oldDisposables ?? [])) {
     try { d.dispose(); } catch (e) {
       console.error(`[PluginHost] Error disposing resource for "${pluginId}":`, e);
     }
   }
   // 从 ResourceTracker 中移除旧 disposable 条目
   // 然后让 ContextBuilder 新注册的 disposables 继续保留
   this.resourceTracker.reap(pluginId, oldDisposables ?? []);
   // ResourceTracker 新增 reap(pluginId, disposables) 方法:
   //   从该 pluginId 的资源列表中移除指定的 disposables
   //   用于部分清理（reload 场景）
   // afterDeactivate 中间件
   const afterDeactCtx = { ...middlewareCtx, phase: 'afterDeactivate' };
   const afterDeactPipeline = compose(this.getMiddleware('afterDeactivate'));
   await afterDeactPipeline(afterDeactCtx, async () => {});
   ```
8. **替换引用**:
   ```
   this.pluginInstances.set(pluginId, newInstance);
   // State 保持 ACTIVE（不需要状态转换）
   ```
9. **更新 DB**:
   ```
   this.db.prepare('UPDATE plugins SET source_code = ?, manifest = ?, updated_at = ? WHERE id = ?')
     .run(newSourceCode, JSON.stringify(newManifest), Date.now(), pluginId);
   ```
10. **失败回滚**: 在步骤 5-6 失败时:
    - 不清理旧版本资源
    - 不调用旧 deactivate
    - 不修改 pluginStates
    - 不修改 pluginInstances
    - 不修改 DB
    - 新激活期间注册的临时 disposables 需要清理（避免 ResourceTracker 泄漏）
      ```
      const newDisposables = diffNewFromOld(oldDisposables, this.resourceTracker.snapshot(pluginId));
      for (const d of newDisposables) { try { d.dispose(); } catch {} }
      this.resourceTracker.reap(pluginId, newDisposables);
      ```
    - 抛出 HotReloadActivationError(pluginId, filePath, cause)

**Verification:**
- 正常 reload 流程: 旧 deactivate 被调用 → 资源清理 → 新实例替换旧实例 → DB 更新
- 激活失败: 旧实例保留 → 状态不变 → DB 不变 → 抛出 HotReloadActivationError
- manifest.id 不一致时抛出明确的错误

### Task 3: 热重载集成测试

**Files:**
- `packages/core/plugin-host/__tests__/plugin-host.test.ts` — 新增 hot reload 测试套件

**Description:**

添加 6 个热重载测试用例:

1. **Reload 成功 — 基本流程**
   - 安装插件 v1 → 激活 → reload(v2) → 确认新版本生效
   - 验证: pluginInstances 指向新实例, DB source_code 已更新

2. **Reload 成功 — 旧 deactivate 被调用**
   - 安装插件（deactivate 记录 side effect）→ activate → reload
   - 验证: 旧 deactivate 被调用一次

3. **Reload 成功 — 旧资源被清理**
   - 安装插件（activate 中注册定时器到 ResourceTracker）→ activate → reload
   - 验证: resourceTracker.disposeAll() 被调用, 资源被 dispose

4. **Reload 失败 — 旧版本保留**
   - 安装插件 v1 → activate → reload(有语法错误的新源码)
   - 验证: 旧实例保持不变, 状态仍为 ACTIVE, 抛出 HotReloadActivationError

5. **Reload 失败 — manifest.id 不匹配**
   - 安装插件 v1 (id=A) → activate → reload(manifest.id=B 的新源码)
   - 验证: 抛出错误包含 "manifest id mismatch"

6. **连续 10 次 Reload — 无泄漏**
   - 安装插件 → activate → 循环 10 次: reload(v1→v2→v3...)
   - 验证: 内存中 pluginStates.size 不增长, pluginInstances.size 不增长
   - 验证: ResourceTracker 中该 pluginId 无残留资源
